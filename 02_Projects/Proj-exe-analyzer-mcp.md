## 0. 개요
- 목표: **실행 파일(Windows PE / ELF / Mach-O 등)** 을 대상으로 정적 분석을 수행하고, 결과를 **MCP 서버 형태로 표준화된 JSON** 으로 제공하여(LLM/에이전트가 소비), 리버스 엔지니어링·보안 점검·의존성/구조 파악 작업을 자동화
- 마감: (대화 내에서 확정된 마감일 언급은 확인되지 않음) → 우선 **“POC 1차”** 기준으로 단계별 목표를 설정

## 1. 로드맵
- [ ] 요구사항 정리
  - [ ] 지원 포맷 범위 확정: PE(우선) → ELF/Mach-O(확장)
  - [ ] “정적 분석만”을 기본 원칙으로(실행/동적분석은 별도 단계)
  - [ ] MCP 인터페이스(메서드/입출력 스키마) 초안 작성
- [ ] 분석 파이프라인/도구 선정 리서치
  - [ ] 1차(가벼운 정적): 해시/메타데이터/섹션/임포트·익스포트/엔트로피/문자열/리소스
  - [ ] 2차(심화): 디스어셈블/함수 경계/심볼/콜그래프(옵션)
  - [ ] 라이브러리 후보(예): LIEF, pefile, pyelftools, macholib, Capstone, radare2, Ghidra headless(선택)
- [ ] POC 아키텍처 설계
  - [ ] MCP 서버(코어) + 분석 어댑터(포맷별) + 결과 정규화(공통 스키마) 계층 분리
  - [ ] 캐시/중복 분석 방지(해시 키), 타임아웃/리소스 제한, 샌드박싱(경로 제한) 설계
- [ ] 최소 기능 구현(MVP)
  - [ ] `analyze(file)` : 파일 식별 + 해시 + 포맷 판별 + 핵심 메타 + 임포트/익스포트 + 섹션 테이블
  - [ ] `strings(file, opts)` : 문자열 추출(ASCII/UTF-16) + 최소 필터링
  - [ ] `ioc(file)` : 엔트로피/패커 의심 지표/서명 여부(가능 시) 등 “휴리스틱 요약”
- [ ] 신뢰성/안전성 보강
  - [ ] 악성 입력 방어(대용량/손상 파일/Zip bomb 유사 케이스), 분석 단계별 타임아웃
  - [ ] 결과 스키마 버전 관리(`schema_version`)
  - [ ] 단위 테스트(샘플 PE/ELF) + 스냅샷 테스트(JSON)
- [ ] 확장 기능(옵션)
  - [ ] 디스어셈블/함수 리스트/콜그래프(도구 의존)
  - [ ] YARA 룰 연동(규칙은 외부 제공, 엔진은 선택)
  - [ ] SBOM/의존 DLL 요약, 취약 라이브러리 탐지(가능한 범위 내)
- [ ] 데모 준비
  - [ ] “하나의 exe를 넣으면 → 구조 요약/의심 지표/의존성/주요 스트링”을 카드 형태로 보여주는 데모 시나리오
  - [ ] mcp-git-analyzer 등과 연계한 “레포 + 바이너리” 통합 분석 흐름 예시
